ice_signin 后续优化记录

当前状态

ice_signin 目前已经有一版可用的保底方案：

  • 文件：/root/.openclaw/workspace-code/ice_signin.py
  • 当前可用模式：browser-checkin
  • 当前结论：
  • 浏览器驱动签到链路已打通
  • 真实签到动作已确认：POST https://signv.ice.v.ua/checkin?next=/embed
  • 纯 requests 方案仍未完全闭环

已完成的关键点

  1. 主站 ice.v.ua 登录态获取已打通
  2. signv.ice.v.ua iframe 真实上下文已定位
  3. 已确认跨域 iframe 是主要难点
  4. 已抓到真实签到动作请求
  5. 已落地浏览器驱动版保底签到逻辑

后续准备优化的方向

1. 提升 browser-checkin 稳定性

目标：让浏览器驱动版更适合长期定时运行。

待做：

  • 自动寻找目标 page，而不是依赖固定页面状态
  • 如果页面不存在则自动新开 tab
  • 自动判断登录是否失效
  • 自动区分：
  • 今日已签到
  • 刚刚签到成功
  • 登录失效
  • 页面结构变化
  • 按钮不存在
  • 输出更稳定的退出码

2. 配置化

目标：减少硬编码，便于复用。

待做：

  • 支持配置文件写入 cdp_url
  • 支持多个页面 URL 候选
  • 支持超时、等待时间、重试次数配置
  • 支持指定远程调试实例

3. 定时执行化

目标：让它能挂 cron / 调度器稳定跑。

待做：

  • 补充更适合 cron 的简洁输出模式
  • 增加日志文件输出选项
  • 增加失败重试机制
  • 增加成功/失败通知接口（后续可接飞书/QQ/其他提醒）

4. 继续尝试压缩成纯脚本版

目标：评估是否能在未来把浏览器依赖降到最低。

当前已知困难：

  • 仅靠 auth_token + user_id + signv session 仍不足以稳定重放
  • signv 很可能依赖真实浏览器上下文 / iframe 链路 / 时序状态

后续可尝试：

  • 更完整抓取签到前后的 cookie 差异
  • 对比浏览器点击前后的请求序列
  • 检查是否存在额外隐藏字段 / CSRF / 中间跳转态
  • 评估是否能用更完整的 HTTP 重放替代浏览器点击

5. 文档与使用说明

目标：以后重新接手时更快恢复上下文。

待做：

  • 补一份使用说明文档
  • 补一份故障排查文档
  • 明确哪些前提必须满足：
  • Chrome 已登录
  • 远程调试口可访问
  • 签到页可打开

当前建议

如果后续重新启动这个项目，优先顺序建议是：

  1. 先把 browser-checkin 做稳
  2. 再做 cron / 通知 / 日志
  3. 最后再尝试纯 requests 版闭环

当前封存结论

ice_signin 项目先暂告一段落。

后续如继续推进，优先按“浏览器驱动保底方案稳定化”这条线做，不建议一上来就重新死磕纯 requests 版。
