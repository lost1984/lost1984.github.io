当前多 Agent 系统工作流与说明书（草案 v1）

一、系统目标

当前这套系统的理想状态，不是让所有 Agent 都能做所有事，而是形成一个稳定分层：

  1. 主 Agent 负责判断与对外接口
  2. 专业 Agent 负责把某类任务做深
  3. 后台/巡检 Agent 负责监控、提醒、状态维护
  4. 文件系统负责长期稳定记忆与工作协议

二、当前建议的 Agent 分层

主控层

  • 主 Agent
  • 面向用户
  • 负责统筹、判断、定优先级

专业执行层

  • 研究/检索 Agent
  • 文档/知识整理 Agent
  • 系统/配置改造 Agent
  • 日历/任务/流程 Agent

后台支持层

  • Heartbeat / 巡检 Agent
  • 记忆维护 Agent
  • 状态汇总与提醒 Agent

三、各 Agent 的职责定义

1. 主 Agent（Main Agent）

主 Agent 的本质：你唯一的正面接口人 + 最终判断者。

主要职责

  • 直接与用户对话
  • 识别任务类型与优先级
  • 决定是否需要调用其他 Agent
  • 汇总多个 Agent 的结果并给出最终输出
  • 在信息不全时提出关键问题，而不是机械追问
  • 把复杂问题拆成阶段性行动方案

不该承担的职责

  • 不应长期陷入大批量机械处理
  • 不应被大量低价值巡检拖住
  • 不应一边当裁判，一边无限细做所有重复劳动

成功标准

  • 输出稳定
  • 上下文连续
  • 能判断，不只是转发结果
  • 用户只需要和它对话，就能推动整个系统

2. 研究 / 检索 Agent

研究 Agent 的本质：外部信息探索器。

主要职责

  • 搜索互联网资料
  • 查人物、公司、行业、事件背景
  • 生成初步事实清单、证据链、来源列表
  • 在模糊问题上先做信息收束，再交给主 Agent 判断

边界

  • 不负责最终拍板
  • 不直接代替主 Agent 输出面向用户的最终结论
  • 不把“搜索结果”直接当成“真实结论”

3. 文档 / 知识整理 Agent

文档 Agent 的本质：把杂乱信息压缩成可复用结构资产。

主要职责

  • 读取长文档、附件、会议材料
  • 提炼结构、论点、关键分层
  • 生成纪要、说明书、分析草案、知识文档
  • 将零散讨论沉淀为文档资产

4. 系统 / 配置改造 Agent

系统 Agent 的本质：你的 Agent 工作区架构师与维修工。

主要职责

  • 修改工作区文件
  • 调整 AGENTS / SOUL / USER / TOOLS / HEARTBEAT / MEMORY
  • 优化工作流、职责划分、记忆结构
  • 排查 OpenClaw、插件、配置层问题

5. 日历 / 任务 / 流程 Agent

流程 Agent 的本质：时间、任务、执行跟踪器。

主要职责

  • 管理日历事件
  • 管理任务与待办
  • 查询忙闲状态
  • 维护与执行进度有关的结构化信息

6. Heartbeat / 巡检 Agent

Heartbeat Agent 的本质：安静但警觉的后台巡逻者。

主要职责

  • 定期检查是否存在需要提醒的信息
  • 发现重要变化时才推送
  • 维持低噪音、高价值的主动提醒机制

绝对原则

  • 没有增量，不要说话
  • 没有紧急性，不要打扰
  • 不重复提醒同一件事
  • 只做有限的高价值巡检

四、推荐工作流

工作流 A：用户直接提问

  • 用户提出需求
  • 主 Agent 识别任务类型
  • 判断是否需要专业 Agent
  • 需要则分派给对应专业 Agent
  • 专业 Agent 返回结构化结果
  • 主 Agent 汇总 / 判断 / 输出

工作流 B：长文档或附件处理

  • 用户提供材料
  • 文档 Agent 读取与提炼
  • 形成结构摘要与核心问题
  • 主 Agent 对照当前上下文
  • 输出结论 / 草案 / 下一步建议

工作流 C：系统改造

  • 发现系统问题或职责不清
  • 系统 Agent 诊断
  • 给出改造草案
  • 用户确认后修改工作区文件

工作流 D：后台巡检

  • Heartbeat 触发
  • 检查少数高价值项目
  • 无重要变化则保持安静
  • 有重要变化则生成简明提醒

五、推荐交接规则

  1. 最终输出尽量回到主 Agent
  2. 专业 Agent 返回结构化结果，不要返回散乱碎片
  3. 不要多 Agent 同时抢同一类决策
  4. 后台 Agent 不要直接占据主对话

六、当前系统最容易出现的问题

  • 职责重叠：多个 Agent 都能分析、都能写、都能拍板
  • 职责过空：名义上有多个角色，但没有明确边界
  • 后台过吵：Heartbeat 高频提醒、泛提醒
  • 主 Agent 过载：所有事都自己做

七、我对你当前系统的建议版职责划分

最小可行结构（MVP）

  • 主 Agent：与用户直接对话、判断任务类型、汇总结果、形成最终答复
  • 研究 Agent：搜索、背景调查、候选信息收束
  • 文档 Agent：读材料、写文档、做知识整理与结构提炼
  • 系统 Agent：改配置、管文件、调整工作流、修系统架构
  • 流程 Agent：日程、任务、截止跟踪
  • Heartbeat Agent：后台巡检、低噪音提醒

八、落地建议

建议按以下顺序推进：

  1. 明确你现在到底有哪些 Agent 在运行
  2. 给每个 Agent 定位一句话职责
  3. 写清楚“不该做什么”
  4. 设计交接格式
  5. 再去补人格、名字、风格

九、下一步可继续扩写的内容

  1. 每个 Agent 的命名与人格定位
  2. 主 Agent 与子 Agent 的消息协议模板
  3. 当前工作区文件如何映射到多 Agent 协作架构

十、结论

多 Agent 系统的关键，不在“角色多”，而在“谁负责判断，谁负责执行，谁负责安静地盯着系统”。
